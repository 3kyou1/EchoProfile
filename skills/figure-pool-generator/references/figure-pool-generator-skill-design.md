# EchoProfile 候选池生成 Skill 设计

日期：2026-04-23
状态：Ready for review

## 1. 背景与目标

`EchoProfile` 已经有两份与人物候选池相关的重要文档：

- `docs/echo-profile-figure-pool-material-guidelines.md`
- `docs/superpowers/specs/2026-04-23-figure-pool-design.md`

前者定义了通用人物候选池的素材规范，后者定义了未来的 `FigurePool` / `FigureRecord` 系统方向。但目前项目运行中的正式数据入口仍以 `src/data/scientistPool.json` 为中心，通用 `FigurePool` 的正式实现尚未落地。

本设计的目标不是立即实现生成器，而是先为一个“项目专用候选池生成 skill”定义清楚边界、工作流和质量标准。等 `FigurePool` 正式实现后，可以直接据此实现 skill，并用它根据用户指定主题自动生成一个可直接使用的初版候选池。

该 skill 的核心目标是：

- 用户给出任意主题
- skill 自动联网检索并筛选代表人物
- skill 自动补全人物核心字段与本地头像资源
- skill 直接产出一份可导入的 `FigurePool` JSON
- skill 同时产出一份 Markdown 说明稿，记录收录逻辑、来源与 warning

## 2. 已确认需求

- 这是一个 `EchoProfile` 仓库内的项目专用 skill，而不是全局通用 skill。
- skill 需要支持“任何用户指定的主题”，而不是只支持科学家、企业家、投资人等固定类别。
- skill 默认通过联网检索来整理候选池，不依赖用户先提供完整人物名单。
- skill 可以主动选择少量高代表性人物作为主题锚点，例如拉马努金、比尔盖茨等。
- 交互采用一步生成，不需要先给出候选名单等待用户确认。
- 默认输出同时包含：
  - 一份可导入的 `FigurePool` JSON
  - 一份配套的 Markdown 说明稿
- 默认候选池规模为 `30-50` 人。
- skill 需要自动下载本地头像资源，而不是只保留外部图片链接。
- 质量策略采用混合模式：
  - 默认尽量保证记录完整、可直接导入
  - 当主题过窄时，允许少量 warning 记录以保持规模
  - 但任何降级都必须在 Markdown 说明稿中显式记录
- 等 `FigurePool` 实现后，生成结果应直接写入应用当前正式读取/导入的位置，而不是只写到临时草稿目录。

## 3. 方案选择

已选方案：方案 1

方案 1 的核心方向：

- 先写一份面向未来 `FigurePool` 的 skill 设计文档
- 先定义 skill 的输入、输出、联网检索流程、下载规则、校验规则和目录约定
- 暂不受当前 `scientistPool.json` 旧入口限制
- 等 `FigurePool` 正式落地后，再基于该设计实现 skill

选择这一方案的原因：

- 可以避免把设计绑死在旧的科学家池路径上
- 可以让 skill 从一开始就直接面向通用人物候选池
- 可以减少未来 `FigurePool` 落地后的返工

## 4. skill 定位

这个 skill 的职责是：

- 接受一个主题
- 产出一个符合 `FigurePool` 规范的“基础候选池初版”

这里的“基础候选池初版”不是空模板，也不是半成品清单，而是尽量满足以下条件的可用产物：

- 主题边界清楚
- 人物规模达到默认目标
- 主要展示字段齐全
- 头像已落地为本地静态资源
- JSON 能直接进入未来候选池导入/读取链路

这个 skill 不负责：

- 设计或实现 `FigurePool` 本身的数据层
- 设计候选池管理 UI
- 替用户做多轮人工编辑
- 维护长期增量同步
- 保证所有人物都达到“百科级资料完整度”

## 5. 典型使用方式

等 skill 真正落地后，预期支持类似以下请求：

- “创建一个数学直觉型人物候选池”
- “创建一个长期主义投资人候选池”
- “创建一个计算机先驱人物候选池”
- “创建一个日本现代设计师候选池”

默认行为：

- 直接开始生成，不先返回 shortlist
- 自动从公开资料中筛选 `30-50` 人
- 自动生成池级描述与人物级内容
- 自动下载本地头像
- 自动生成 JSON 与说明稿

建议保留的可选参数：

- 主题名
- 目标规模
- 语言覆盖要求
- 是否强制包含某些锚点人物
- 是否排除某些人物
- 严格模式或宽松模式覆盖
- 是否允许只生成文案而暂不下载头像

这些参数不影响默认路径，但可以在实现阶段作为高级控制项补充。

## 6. 输出产物

### 6.1 主产物：`FigurePool` JSON

这是未来真正用于应用读取、导入或初始化候选池的正式数据。

设计要求：

- 结构面向未来 `FigurePool`
- `records` 内每个对象面向未来 `FigureRecord`
- 默认不写“临时草稿字段”污染正式 schema
- 与运行时强耦合的调试信息尽量放入 Markdown 说明稿，而不是 JSON 主体

### 6.2 附产物：Markdown 说明稿

说明稿用于记录本次候选池生成的上下文与质量判断，便于后续人工复查。

至少应包含：

- 主题名称
- 候选池名称与描述
- 收录逻辑
- 人物总数
- 有效记录数
- warning 记录数
- 未收录但被考虑过的典型人物
- 主要来源链接
- 头像来源与下载结果摘要
- 降级策略说明

### 6.3 静态资源

需要为池内人物下载并保存本地头像资源。

资源要求：

- 路径稳定
- 命名与 `slug` 对齐
- 可被前端静态访问
- 同一候选池内格式和命名风格尽量统一

## 7. 面向未来 schema 的约束

### 7.1 `FigureRecord`

skill 生成的人物记录应至少对齐当前素材规范中的核心字段：

- `slug`
- `name`
- `localized_names`
- `portrait_url`
- `quote_zh`
- `quote_en`
- `core_traits`
- `thinking_style`
- `temperament_tags`
- `temperament_summary`
- `loading_copy_zh`
- `loading_copy_en`
- `bio_zh`
- `bio_en`
- `achievements_zh`
- `achievements_en`

设计原则：

- 尽量产出完整记录，而不是只产出最小可运行字段
- 字段写法遵循 `docs/echo-profile-figure-pool-material-guidelines.md`
- 不因人物类别不同而改变基础 schema

### 7.2 `FigurePool`

skill 生成的池级对象应对齐未来候选池模型，至少考虑这些字段：

- `id`
- `name`
- `description`
- `records`
- `origin`
- `isDefault`
- `createdAt`
- `updatedAt`
- `schemaVersion`
- `validationSummary`

其中：

- `origin` 对 skill 生成的池应有清晰来源语义，例如 `generated` 或后续统一到项目正式值
- `validationSummary` 应反映结构错误、warning 和可用记录数量
- 最终字段命名以 `FigurePool` 正式实现为准

## 8. 生成工作流

### 8.1 主题归一化

输入主题后，skill 先完成主题归一化：

- 抽取主题名称
- 判断人物类型边界
- 推断可能的时代、地区、行业或方法论范围
- 生成候选池名称、描述和内部 `id`

目标是避免得到过宽、过散或无法稳定检索的主题。

### 8.2 候选人物发现

skill 需要通过联网检索形成较大的候选人物集合，然后再收敛到正式入池名单。

候选来源可以包括：

- 官方机构或大学页面
- 传记资料页
- Wikipedia 及对应语种条目
- Britannica 等通用参考来源
- 公司、基金会、研究机构官方人物页

这一阶段的目标不是立刻写字段，而是确保：

- 人物与主题强相关
- 候选集合足够大
- 主题内部有代表性和多样性

### 8.3 代表性筛选

从候选集合进入正式池时，skill 应优先保留：

- 对该主题具有高识别度的人物
- 能代表不同子方向的人物
- 能支撑稳定双语资料补全的人物
- 能获得较稳定单人肖像资源的人物

应避免：

- 同类型人物过度堆叠
- 只因为资料容易找就收录与主题边界弱相关的人物
- 为凑数而引入明显边缘人物

### 8.4 人物字段补全

对于每位入选人物，skill 需要按素材规范生成或整理字段。

重点要求：

- `name` 与 `localized_names` 尽量采用公开稳定名称
- `quote_zh` / `quote_en` 优先选独立可读、能体现思维气质的短句
- `bio_zh` / `bio_en` 不是百科流水账，而是展示导向简介
- `achievements_zh` / `achievements_en` 保持 `2-4` 条高识别度成果
- `core_traits`、`thinking_style`、`temperament_tags`、`temperament_summary` 要能解释“为什么这个人适合人物回响”
- `loading_copy_zh` / `loading_copy_en` 保持轻量且贴近人物气质

### 8.5 头像检索与下载

每位人物都需要匹配一张本地头像。

头像处理流程：

- 检索可公开访问的单人肖像
- 优先选择适合 `4:5` 竖版裁切的图
- 避免群像、强遮挡、低清晰度图片
- 下载到项目静态资源目录
- 统一文件命名并写回 `portrait_url`

头像要求仍以 `docs/echo-profile-figure-pool-material-guidelines.md` 中“照片要求”为准。

### 8.6 结构校验与质量分级

在写入正式输出前，skill 需要对整池执行校验，并按问题严重度分级。

建议分为三层：

- 结构错误：不能进入正式 JSON
- 展示质量问题：可以进入，但需要 warning
- 可降级问题：允许回退，不阻塞导入

### 8.7 写出结果

生成流程完成后，skill 需要统一写出：

- 正式 `FigurePool` JSON
- Markdown 说明稿
- 本地头像资源

未来实现阶段再根据 `FigurePool` 正式落地情况确定最终目录。

## 9. 质量标准与降级策略

### 9.1 默认目标

默认目标不是“尽快凑够人数”，而是“尽量得到一份可直接使用的完整池”。

因此默认策略应为：

- 先保证硬字段完整
- 再追求 `30-50` 人规模
- 再追求覆盖面和丰富度

### 9.2 硬错误

以下问题应视为硬错误，默认不应进入正式候选池：

- `slug` 缺失或重复
- `name` 缺失
- `portrait_url` 缺失或本地资源不可用
- `quote_zh` / `quote_en` 缺失
- `bio_zh` / `bio_en` 缺失
- `achievements_zh` / `achievements_en` 缺失、为空或结构错误

### 9.3 warning

以下问题可以进入池，但必须记录 warning：

- 某些本地化名只能回退到主名
- 标签写法质量一般但不影响展示
- 名言可信度一般，需要后续人工复核
- 简介或成就写法略显平，但不影响卡片展示
- 头像质量达标但不够理想

### 9.4 主题过窄时的混合策略

当主题本身过窄、但用户仍要求默认规模时，skill 可以采用混合策略：

- 仍然禁止硬错误记录进入正式池
- 允许少量 warning 记录进入池中
- 在 Markdown 说明稿中清楚标记 warning 人物与原因
- 说明“规模优先于完美完整度”的本次特例

也就是说，混合模式放宽的是展示质量层，而不是结构正确性层。

## 10. 说明稿内容建议

配套 Markdown 说明稿建议采用固定结构，便于后续人工审阅。

建议至少包含：

### 10.1 池概览

- 主题
- 候选池名称
- 候选池描述
- 记录总数
- 有效记录数
- warning 记录数

### 10.2 收录逻辑

- 主题边界如何定义
- 为什么这些人物适合收录
- 是否有子方向配平

### 10.3 来源摘要

- 主要信息来源类型
- 关键人物的主要来源链接
- 名言与头像的主要来源记录方式

### 10.4 风险与后续人工建议

- 哪些人物建议优先复核
- 哪些字段较弱
- 哪些头像只是“可用”而非“理想”
- 是否建议后续拆分成多个池

## 11. 目录与落位策略

由于 `FigurePool` 还未正式实现，本设计不把最终目录写死，但建议先遵守以下原则：

- 正式 JSON 与运行时正式数据目录同级
- 头像资源进入前端静态资源目录
- 说明稿进入项目文档或生成记录目录
- 不把正式池和临时抓取缓存混放

未来实现时建议至少区分：

- 正式候选池数据
- 正式静态头像资源
- 生成说明稿
- 抓取或中间过程缓存

最终目录命名以 `FigurePool` 落地后的项目结构为准。

## 12. 实现前置条件

在真正实现这个 skill 之前，需要先满足以下条件：

- `FigurePool` / `FigureRecord` 正式 schema 已落地
- 应用当前正式读取/导入的候选池目录已明确
- 头像静态资源正式目录已明确
- 导入校验器或最小校验规则已明确
- 项目对 skill 所在目录和触发方式已有定论
- 网络检索与图片下载所需权限可用

## 13. 风险与待决问题

### 13.1 schema 仍可能调整

当前文档只能面向已知设计做约束，正式实现时应以最终 `FigurePool` schema 为准。

### 13.2 图片版权与可用性

skill 可以下载公开可访问图片，但项目是否需要进一步记录版权、来源、许可信息，仍需在实现阶段明确。

### 13.3 多语言质量波动

不同主题下，中英双语资料完整度差异可能很大。实现阶段需要明确：

- 哪些字段允许翻译生成
- 哪些字段必须基于可靠原始来源整理
- 名言是否允许只找到单语原文后再做翻译

### 13.4 主题宽窄不一

“任何用户指定主题都可以”意味着 skill 必须处理极宽和极窄两类主题。实现阶段需要加入更明确的主题收敛逻辑，以避免得到质量失控的候选池。

## 14. 后续落地顺序

建议后续按以下顺序推进：

1. 完成 `FigurePool` 与 `FigureRecord` 正式实现
2. 明确正式数据目录、头像目录和导入校验接口
3. 在仓库内创建项目专用 skill
4. 将本设计中的工作流落成脚本、参考文档和 `SKILL.md`
5. 用一个真实主题进行端到端试跑
6. 根据试跑结果再收紧校验、warning 分级和说明稿格式

## 15. 结论

本设计定义了一个未来可落地的 `EchoProfile` 项目专用候选池生成 skill：

- 输入是用户指定主题
- 输出是正式 `FigurePool` JSON、配套 Markdown 说明稿和本地头像资源
- 默认一步生成，默认规模为 `30-50` 人
- 默认联网检索并自动筛选代表人物
- 默认追求可直接使用，但在主题过窄时允许少量 warning

在 `FigurePool` 正式落地之前，这份文档作为 skill 的实现规格存在；等相关数据结构和目录稳定后，即可据此进入 skill 创建与实现阶段。

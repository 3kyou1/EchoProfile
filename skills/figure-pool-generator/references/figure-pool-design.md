# EchoProfile 多候选池 Thought Echoes 设计

日期：2026-04-23
状态：Ready for review

## 1. 背景与目标

当前 Thought Echoes 功能依赖固定的 `scientistPool.json`，底层类型、服务和结果缓存均以 `scientist` 为中心建模。这一结构不适合支持用户自定义的多候选池，例如科学家、企业家、投资人或其他任意人物集合。

本设计的目标是将 Thought Echoes 从“固定科学家池”升级为“可导入、可管理、可切换的通用人物候选池系统”，并确保历史结果与生成时所使用的候选池保持稳定绑定。

## 2. 已确认的产品需求

- Thought Echoes 支持多个候选池，而不是单一固定池。
- 用户可以维护多个候选池，并自由命名。
- 首版优先支持整池导入 JSON。
- 后续也支持在应用内逐个编辑人物。
- 导入后的候选池复制进应用内部存储，不依赖原始外部文件。
- 应用内存在候选池列表，供用户选择本次生成所使用的候选池。
- 候选池选择绑定到某次结果；每次生成前选择一个池，结果永久记住当时所用池。
- 切换候选池时，旧结果保留，但需要明确显示其来源池。
- 候选池管理首版支持：导入、查看基本信息、选择用于本次生成、删除、重命名、编辑池内人物、导出、设为默认。
- 交互结构采用两层：
  - Thought Echoes 页面提供候选池选择器和快捷入口。
  - 详细管理进入独立的候选池管理页面。
- 导入校验失败时不直接拒绝；合法人物立即可用，错误人物保留在池内但不参与匹配，供用户后续修正。
- 内置候选池与自定义候选池权限一致，都可以改、删、重命名。
- 首版即泛化底层数据模型，不再把系统限定为 `scientist` 语义。
- 人物数据采用完整 schema，要求与当前 `scientistPool.json` 同等级别的信息完整度。

## 3. 方案选择

已选方案：方案 B

方案 B 的核心方向：

- 首版即把 Thought Echoes 从固定科学家池升级为通用人物候选池系统。
- 底层类型、服务、存储和结果缓存都泛化为可支持任意人物主题的模型。
- 内置科学家池只是默认内置池之一，而不是特殊写死逻辑。

## 4. 数据模型与存储

### 4.1 人物记录模型

建议把现有 `ScientistRecord` 泛化为 `FigureRecord`。

字段原则：

- 尽量沿用当前完整 schema，避免损失已有卡片展示能力。
- 同一套 schema 同时支持科学家、企业家、投资人等不同类型人物池。

建议字段：

- `slug`
- `name`
- `localized_names`
- `portrait_url`
- `quote_zh`
- `quote_en`
- `core_traits`
- `thinking_style`
- `temperament_tags`
- `temperament_summary`
- `loading_copy_zh`
- `loading_copy_en`
- `bio_zh`
- `bio_en`
- `achievements_zh`
- `achievements_en`

### 4.2 候选池模型

建议新增 `FigurePool` 作为候选池实体。

建议字段：

- `id`
- `name`
- `description`
- `records`
- `origin`：`builtin` | `imported`
- `isDefault`
- `createdAt`
- `updatedAt`
- `schemaVersion`
- `validationSummary`

设计意图：

- 内置池与导入池统一建模。
- 默认池能力通过字段显式表达，不依赖写死逻辑。
- 池是否可用主要由池内人物的校验结果决定，而不是简单的池级状态切换。

### 4.3 回响结果模型

建议把现有 `ScientistResonanceResult` 泛化为 `FigureResonanceResult`。

在现有结果结构上新增池快照信息：

- `pool_id`
- `pool_name_snapshot`
- `pool_updated_at_snapshot` 或等价版本快照字段

设计意图：

- 每条回响结果都明确绑定生成时使用的候选池。
- 即使候选池后续被改名、编辑或删除，旧结果仍然能说明其来源。

### 4.4 存储拆分

建议使用两个 store。

#### `figure-pools.json`

负责：

- 候选池列表
- 默认池信息
- 人物级校验状态与错误摘要

#### `figure-resonance.json`

负责：

- Thought Echoes 生成结果
- 与候选池绑定的缓存记录

拆分原因：

- 候选池管理和结果缓存职责不同。
- 删除、导出、编辑候选池时不应直接耦合结果数据。
- 更容易处理“原池已删除但旧结果保留”的情况。

### 4.5 人物级校验与容错

校验粒度从“整池”下沉为“单个人物”。

建议新增人物级校验状态：

- `valid`
- `invalid`

行为规则：

- 某个人物存在硬错误时，仅该人物标记为 `invalid`。
- `invalid` 人物不参与匹配。
- `invalid` 人物不出现在生成结果中。
- 同池内其他 `valid` 人物继续可用。

导入时的规则：

- 合法人物立即可用。
- 有错误的人物一并导入，但标记为 `invalid`。
- 不再因为少数错误人物阻塞整个候选池。

典型硬错误示例：

- 某人物缺少 `portrait_url`
- `achievements_zh` 不是数组
- `slug` 重复
- 必填字段为空
- 用于匹配的关键字段为空，导致无法安全参与匹配

### 4.6 迁移策略

首版迁移建议：

- 将 `src/data/scientistPool.json` 视为内置默认池的种子数据。
- 首次运行或新 store 为空时，自动初始化一个内置候选池。
- 迁移后为每个人物建立校验状态；默认内置数据应全部初始化为 `valid`。
- 旧的回响结果缓存不强制迁移，可在需要时重新生成。

迁移原则：

- 优先保证新候选池系统结构稳定。
- 降低首版一次性迁移旧缓存的复杂度。

## 5. 页面结构与主流程

### 5.1 页面职责拆分

采用两层结构。

#### Thought Echoes 页面

负责：

- 选择本次生成要使用的候选池
- 发起生成
- 查看当前与历史回响结果
- 通过快捷入口进入候选池管理或导入流程

设计原则：

- 让“选池再生成”成为主路径。
- 不把复杂管理操作堆在结果页里。

#### 候选池管理页面

负责：

- 查看候选池列表
- 导入候选池
- 重命名、删除、导出、设默认
- 进入某个候选池的详情与人物编辑
- 修正无效人物

设计原则：

- 把完整管理能力集中在独立页面。
- 与生成流程解耦，避免结果页承担过多编辑逻辑。

### 5.2 Thought Echoes 页面主流程

建议主流程：

1. 用户进入 Thought Echoes 页面。
2. 系统根据默认池预选一个候选池。
3. 用户可以切换本次要使用的候选池。
4. 点击生成。
5. 系统使用当前选中池的人物集合进行匹配。
6. 结果写入缓存，并绑定池快照信息。

### 5.3 结果绑定与历史保留

每次生成结果都需要显式绑定候选池。

结果需记录：

- `pool_id`
- `pool_name_snapshot`
- 生成时间

行为规则：

- 切换候选池不会清空旧结果。
- 旧结果继续保留并显示其来源池。
- 用户可在新池上再次生成，形成新的结果记录。

### 5.4 默认池语义

默认池的职责限定为：

- 影响下次进入页面或发起生成时的预选值

默认池不应：

- 覆盖历史结果的池绑定
- 强制替换用户当前手动选择的池

### 5.5 无效人物与生成隔离

正式生成流程按人物级状态过滤。

行为规则：

- 生成时仅使用 `valid` 人物。
- `invalid` 人物在管理页中可见、可编辑，但不会进入匹配链路。
- 候选池本身仍可被选择和生成，只要其中存在可用人物。

### 5.6 候选池选择器提示

当候选池中存在 `invalid` 人物时，Thought Echoes 页的候选池选择器直接显示可用性摘要。

展示形式示例：

- `企业家池（17 可用 / 3 无效）`

补充提示：

- 生成按钮附近可增加轻提示，说明系统会自动跳过无效人物。

### 5.7 当前查看结果与下次生成池分离

结果页需要显式区分两个概念：

- 当前正在查看的结果
- 当前为下次生成所选择的候选池

允许出现以下状态：

- 用户正在查看基于 A 池生成的历史结果。
- 候选池选择器已切换到 B 池，准备下次生成。

建议提示方式：

- 在结果区显示：`当前正在查看：科学家池的历史结果`
- 在生成区显示：`当前已选择：企业家池，用于下次生成`

## 6. 候选池管理页与人物编辑器

### 6.1 管理页整体布局

建议采用两栏结构。

#### 左栏：候选池列表

每个候选池展示：

- 名称
- 人物数量
- 校验摘要：例如 `12 可用 / 3 无效`
- 来源：`builtin` | `imported`
- 是否默认

顶部主操作：

- `导入候选池`
- `新建候选池`

每个候选池的快捷操作：

- 重命名
- 导出
- 设默认
- 删除

#### 右栏：当前选中池详情

建议展示：

- 候选池名称
- 描述
- 更新时间
- 校验状态
- 池内人物列表
- 池级主操作入口

### 6.2 池内人物列表

首版建议使用管理导向的列表或表格视图，而非重卡片视图。

每条人物至少展示：

- 姓名
- `slug`
- 头像缩略图
- 校验状态
- 最近更新时间

顶部操作：

- `新增人物`
- `批量校验`
- `搜索人物`

行内操作：

- 编辑
- 复制
- 删除

### 6.3 人物编辑器形态

人物编辑器采用右侧抽屉或侧边面板，而不是独立详情页。

原因：

- 编辑时可继续对照当前池的其他人物。
- 更适合“修完一个错误继续下一个”的连续操作。
- 与两栏管理布局配合更自然。

### 6.4 人物编辑器字段分组

建议按以下区块组织。

#### 基础信息

- `slug`
- `name`
- `localized_names`

#### 媒体信息

- `portrait_url`

#### 匹配信息

- `core_traits`
- `thinking_style`
- `temperament_tags`
- `temperament_summary`

#### 展示文案

- `quote_zh`
- `quote_en`
- `loading_copy_zh`
- `loading_copy_en`

#### 内容信息

- `bio_zh`
- `bio_en`
- `achievements_zh`
- `achievements_en`

### 6.5 校验异常视图增强

当候选池中存在 `invalid` 人物时，右栏顶部增加校验状态条。

建议能力：

- 显示错误总数
- 显示受影响人物数
- 提供筛选：
  - 全部人物
  - 仅看有错误的人物
- 提供操作：
  - `重新校验`
  - `仅显示可参与匹配人物`

## 7. 候选池生命周期流程

### 7.1 导入流程

建议流程：

1. 用户在 Thought Echoes 页面或管理页点击 `导入候选池`。
2. 选择 JSON 文件。
3. 系统解析并执行 schema 校验。
4. 合法人物写入为 `valid`。
5. 错误人物写入为 `invalid`。
6. 展示导入结果摘要。

建议导入结果摘要包含：

- 池名
- 总人物数
- 可用人物数
- 无效人物数
- 错误摘要

### 7.2 编辑流程

行为规则：

- `valid` 和 `invalid` 人物都允许编辑。
- 每次保存后重新校验当前人物。
- 若错误修复成功，则该人物从 `invalid` 变为 `valid`。
- 若编辑后引入硬错误，则该人物从 `valid` 变为 `invalid`。

### 7.3 导出流程

导出内容应为完整 schema 的内容包，而不是内部数据库快照。

导出应包含：

- 候选池基础信息
- 人物内容数据

导出不应包含：

- `isDefault`
- 内部校验状态
- 结果缓存关联信息

### 7.4 删除流程

删除候选池时：

- 删除池本身
- 不删除基于该池生成的历史结果
- 历史结果显示 `原候选池已删除` 或等价提示

### 7.5 设默认流程

默认池只影响未来的预选值。

行为规则：

- 设置默认池不会影响历史结果。
- 设置默认池不会自动触发重新生成。
- 若默认池被删除，系统自动回退到最近一个可用池。

## 8. 结果展示与历史结果

### 8.1 结果元信息

每条 Thought Echoes 结果都需要附带固定元信息。

建议展示：

- 来源候选池名称
- 生成时间
- 池快照状态

池快照状态示例：

- `当前池已更新`
- `原候选池已删除`

### 8.2 历史结果列表

建议结果区提供简洁的历史结果列表。

每条至少展示：

- 候选池名
- 生成时间
- 长期主回响人物
- 是否包含 recent state

### 8.3 当前查看结果与当前选择池的双状态提示

界面应同时表达：

- 当前正在查看哪一次结果
- 当前已为下次生成选中了哪个池

建议文案方向：

- `当前正在查看：科学家池的历史结果`
- `当前已选择：企业家池，用于下次生成`

## 9. 底层重构范围

首版即泛化底层，不只调整显示文案。

建议重构范围：

### 9.1 类型层

- `ScientistRecord` -> `FigureRecord`
- `ScientistResonanceCard` -> `FigureResonanceCard`
- `ScientistResonanceResult` -> `FigureResonanceResult`

### 9.2 服务层

- `src/services/scientistResonanceService.ts` -> `src/services/figureResonanceService.ts`
- 固定 `scientistPool.json` 的读取逻辑改成按当前候选池读取 `records`

### 9.3 数据层

- `src/data/scientistPool.json` 从“唯一候选池”降级为“内置 seed pool 数据来源”
- 首次初始化时导入为内置默认池

### 9.4 组件层

- `ScientistResonanceCard.tsx` 改为更通用的人物回响卡组件
- 用户可见文案不再预设“科学家”

### 9.5 术语层

- 用户可见文案统一为“候选池”“人物回响”或等价更通用表述
- 避免继续使用 `scientist` 作为产品层术语

## 10. 迁移与测试策略

### 10.1 迁移重点

- 空 store 首次启动时，自动生成内置默认池
- 旧用户升级后，即使新 store 缺失，也不会崩溃
- 旧结果缓存允许不迁移，必要时重新生成

### 10.2 校验与导入测试

- 全合法 JSON：整池立即可用
- 部分非法 JSON：合法人物可用，非法人物标记为 `invalid`
- 重复 slug、类型错误、缺字段都能准确报出

### 10.3 匹配过滤测试

- 生成时只使用 `valid` 人物
- `invalid` 人物绝不进入 long-term 或 recent-state 结果
- 候选池只有部分合法人物时，仍可正常生成

### 10.4 结果绑定测试

- 同一个 profile 在不同池下生成，应得到不同缓存记录
- 切换池不会覆盖旧结果
- 删除池后，旧结果仍可显示，只是标记来源池已删除

### 10.5 缓存键要求

缓存 key 至少包含：

- `scope`
- `profileId`
- `language`
- `poolId`

这样可避免不同候选池下的结果互相覆盖。
